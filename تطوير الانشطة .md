# Activities Module Closure Plan — Final

## Summary
نبدأ برنامج إغلاق موديول الأنشطة بمنهج **Operational-first** مع **ضم فجوات الأهداف الآن**.  
الهدف هو إغلاق الفجوات التي تمنع الموديول من أن يكون منتجًا ناضجًا ومترابطًا: اكتمال النشاط، تفعيل checklists وtemplates، تصحيح الدقة في الصلاحيات والتنقل، ثم إغلاق target editing gap، وبعدها فقط structural cleanup.

التقييم التنفيذي الحالي:
- البنية: قوية جدًا
- المنطق التشغيلي: قوي
- النضج المنتجـي: غير مغلق بعد
- الأولوية: **إكمال الأسطح التشغيلية قبل cleanup البنيوي**

## Wave B — Targets UX Closure Inside Activities
تأتي مباشرة بعد Wave A، وما زالت ضمن نفس برنامج الإغلاق الحالي.

### 1. Target Edit Strategy
لن نعيد بناء [TargetForm.tsx](/C:/Users/HP/OneDrive/Desktop/new-edara-sys/src/pages/activities/TargetForm.tsx) كـ full edit wizard الآن.  
القرار المعتمد:
- يبقى `TargetForm` **Create-first**
- يتم إغلاق فجوة التعديل عبر **Target Settings / Metadata Edit surface** داخل أو من [TargetDetail.tsx](/C:/Users/HP/OneDrive/Desktop/new-edara-sys/src/pages/activities/TargetDetail.tsx)

### 2. What Becomes Editable
يُضاف مسار تعديل محترف للحقول غير المغطاة حاليًا:
- الاسم
- الوصف
- الملاحظات
- بعض الحقول الفلترية/النطاقية التي يمكن تعديلها بأمان
- مع حسم واضح لما هو:
  - editable post-creation
  - locked post-creation
- adjustments الرقمية الحالية تبقى كما هي إذا كانت already correct

### 3. Route Consistency
- route `/activities/targets/:id/edit` لا تبقى redirect شكليًا
- إما تتحول إلى surface تعديل حقيقية
- أو تُزال ويصبح التعديل من داخل التفاصيل فقط بشكل صريح
- لا نترك UX توحي بوجود edit path ثم تعيد المستخدم بلا فائدة

### 4. Target Wizard Note
صلابة الـ wizard الحالية في `TargetForm` مقبولة مؤقتًا ما دامت:
- create flow تعمل جيدًا
- edit gap أُغلق عبر surface مستقلة
أما مرونة القفز بين الخطوات داخل الـ wizard نفسها فهي **مؤجلة** وليست blocker في هذه المرحلة

## Assumptions And Defaults
- تم اعتماد **Operational-first** بدل structural-first
- تم اعتماد **ضم مشاكل الأهداف الآن** بدل تأجيلها
- checklist management ستكون داخل موديول الأنشطة نفسه
- template activation ستكون داخل الرحلات الحالية، لا كنظام منفصل ضخم
- Target edit سيُحل بمسار metadata/settings focused، لا بإعادة بناء `TargetForm` كاملة
- `subject` / `subject_type` مؤجلة لأنها مرتبطة بكيانات غير مبنية بعد
- `attachments` مؤجلة كميزة لاحقة تعتمد على storage/upload surface
- offline/PWA وPDF/reporting الثقيل خارج النطاق الحالي
